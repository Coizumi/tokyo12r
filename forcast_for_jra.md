# JRA Forecast Logic

対象実装: `scripts/jra_site_updater.py`  
確認時点: `main`

## 目的

TOKYO12R by ZIN のJRA予想ロジックは、JRA公式の出走表HTMLを内部処理し、各レースの上位5頭に予想印を付けるための決定的なヒューリスティックです。

機械学習モデル、外部オッズ、天候、馬場状態、調教、コメント、新聞印は現在使用していません。血統は `data/Sire_data.csv` を静的参照し、全レースでレース条件適性の補正に使います。

## 入力データ

`--fetch-official` 実行時に、JRA公式サイトの `JRADB/accessD.html` へPOSTして以下を取得します。

- 開催一覧
- レース一覧
- 各レースの出走表詳細

対象日でレースが見つからない場合は、対象日から最大4日先までを探索し、最初に取得できた開催日を採用します。

各馬について内部的に使う項目は以下です。

- 馬番
- 馬名
- 性齢
- 騎手
- 獲得賞金
- 父名
- 母の父名
- 近走4走分のテキスト

公開JSONおよびHTMLには、馬番、騎手、成績詳細、獲得賞金、過去走詳細、スコアは出しません。公開対象は予想印、馬名、人気表示、買い目に限定します。

## 基礎スコア計算

各馬に対して `score_horse()` が基礎スコアを計算します。近走4走が全馬分そろわないレースでは、この基礎スコアに血統レース条件適性を加えて印を決めます。最終スコアは小数第3位で丸めます。

### 1. 獲得賞金

獲得賞金が取得できる場合、以下を加点します。

```text
log10(獲得賞金[円]) * 2.1
```

賞金表記は `億` または `万` を円に換算します。

### 2. 近走4走

近走は新しい順に以下の重みを掛けます。

```text
1走前: 1.00
2走前: 0.72
3走前: 0.52
4走前: 0.36
```

着順と頭数が取れる場合、以下を加点します。

```text
max(0, (頭数 + 1 - 着順) / 頭数) * 8.0 * 近走重み
```

3着以内の場合は追加加点します。

```text
(4 - 着順) * 1.4 * 近走重み
```

人気が取れる場合、以下を加点します。

```text
max(0, 18 - 人気) * 0.12 * 近走重み
```

括弧内の数値を着差として扱います。数値が0以上の場合のみ、以下を加点します。

```text
max(0, 2.0 - 着差) * 0.75 * 近走重み
```

### 3. 馬番による微小なタイブレーク

最後に馬番でごく小さな補正を入れます。

```text
(19 - 馬番) * 0.015
```

これは実質的に同点時の並びを安定させるための補正です。

## 追加指数

`make_feature_picks()` またはfallback側の印決定で以下の指数を使います。持ちタイム、末脚、先行力は全馬の近走4走がそろう場合に計算します。血統レース条件適性は、新馬戦などの戦績がない馬も補正できるように全レースで計算します。各指数はレース内または式上で0から100に正規化します。値が取れない場合は中立値の50です。

### 持ちタイム指数

過去走の距離と走破時計から速度を計算し、近走重み付き平均を作ります。

```text
speed = 距離[m] / 走破秒
time_index = レース内で0から100に正規化したspeed
```

走破時計は `1:46.8` のような分秒表記に加え、1000m戦などの `56.4` 形式も対象にします。

### 末脚指数

JRA出走表の近走テキストから、着順品質と通過順からの押し上げを使います。

```text
finish_quality = (頭数 + 1 - 着順) / 頭数
gain = max(0, 4角通過順 - 着順) / 頭数
closing_raw = finish_quality * 0.65 + gain * 0.35
closing_index = レース内で0から100に正規化したclosing_raw
```

上がり3Fが馬柱テキストから安定取得できない場合があるため、現行実装では通過順からの押し上げと着順品質を中心にします。

### 先行力指数

取得できる最初の通過順が前にいるほど高くなるように計算します。

```text
early_position_score = 1 - (最初の通過順 - 1) / max(頭数 - 1, 1)
pace_index = レース内で0から100に正規化したearly_position_score
```

### 血統レース条件適性

父名と母の父名を `data/Sire_data.csv` に照合し、レース条件の芝/ダートと距離に合うほど高くします。父の適性を基本値にし、母の父は中立値を超える分だけ補助加点します。

この指数は全レースに適用します。新馬戦、転入初戦、近走4走が不足する馬では近走由来の評価が横並びになりやすいため、基礎スコアに以下の補正を加えます。

```text
fallback_score = base_score + sire_fit_score * 0.08
```

CSV列は以下です。

- `sire_name`: 種牡馬名
- `surface_axis`: `-100=ダート寄り`, `0=中間`, `100=芝寄り`
- `distance_m`: 適性距離の中心値
- `confidence`: 数値化の信頼度
- `source`: 参照元

レース条件の芝/ダートは以下に変換します。

```text
芝     => 100
ダート => -100
不明   => 0
```

父および母の父それぞれの単体適性は以下で計算します。

```text
surface_fit = 1 - min(abs(stallion.surface_axis - race_surface_axis), 200) / 200
distance_fit = 1 - min(abs(stallion.distance_m - race_distance_m), 600) / 600
lineage_fit_score = (surface_fit * 0.55 + distance_fit * 0.45) * 100
```

最終的な血統レース条件適性は以下です。

```text
sire_fit_score =
  min(100, sire_lineage_fit_score + max(0, dam_sire_lineage_fit_score - 50) * 0.35)
```

父名または母の父名がCSVにない場合、または距離が取れない場合は、その系統の単体適性を中立値の `50` とします。母の父が中立値以下の場合は減点せず、父の適性だけを使います。

### 総合力指数

基礎スコアをレース内で0から100に正規化し、追加指数を小さく加味して総合力ランキングを作ります。

```text
overall_rank_score =
  overall_index
  + time_index * 0.10
  + closing_index * 0.08
  + pace_index * 0.06
  + sire_fit_score * 0.08
```

## 印の決定

### 全馬の近走4走がそろう場合

印は以下のルールで決めます。

```text
◎: 持ちタイム指数 + 末脚指数の最優秀馬
○: 持ちタイム指数 + 先行力指数の最優秀馬
▲: 持ちタイム指数 + 末脚指数の2番手
△: 総合力指数の最優秀馬
☆: 血統レース条件適性の最優秀馬
```

同じ馬が複数の印候補になる場合は、その印のランキングで次に高い未選出馬を繰り上げます。例えば◎と○が同じ馬になる場合、○は `持ちタイム指数 + 先行力指数` の次点馬になります。▲も同じく、◎で選ばれていない次点馬を採用します。

同点時は馬番昇順、馬名昇順で並びを安定させます。

### 近走4走がそろわない馬がいる場合

1頭でも近走4走がそろわない場合は、基礎スコアに血統レース条件適性を加えたfallback方式にします。新馬戦など、賞金や近走がほぼ空の馬が多いレースでも、レース条件に合う血統を少し上に置けるようにします。

```text
1. 基礎スコア + sire_fit_score * 0.08 の降順
2. 馬番昇順
3. 馬名昇順
```

上位5頭に `◎`, `○`, `▲`, `△`, `☆` を順に割り当てます。

5頭未満しか印が作れない場合、買い目は生成しません。

## 買い目生成

印が5頭そろった場合、`bet_sections()` が固定フォーメーションで買い目を作ります。

生成対象は次の3種類です。

- 馬連フォーメーション
- 3連複BOX
- 3連単フォーメーション

`穴狙い馬単` は生成しません。3連単フォーメーションが的中する並びでは、☆1着固定の馬単は必ず不的中になり、結果ページの視認性と回収率集計の見え方を悪くするためです。

### 馬連フォーメーション

```text
◎○ - ○▲△☆
```

重複を除外し、合計7点です。

### 3連複BOX

```text
◎○▲△☆
```

5頭BOXのため、合計10点です。

### 3連単フォーメーション

```text
◎○ - ◎○▲ - ◎○▲△☆
```

同一馬の重複を除外し、合計12点です。

## 公開データの削減

内部処理ではスコア、成績、過去走、賞金、馬番、騎手を使いますが、`public_payload()` では以下のみを書き出します。

- 開催日
- 生成時刻
- データソース名
- 会場
- レース番号
- 発走時刻
- レース名
- コース
- 予想印、馬名、人気表示
- 買い目

これにより、出走表や馬柱の再掲にならないようにしています。

## 更新頻度

サイト更新は、JRA公式HTMLを取得して内部処理し、公開用HTML/JSONだけをCloudflare Pagesへ反映する想定です。

基本スケジュールはJSTで以下です。

```text
金曜 22:00
土曜 09:32, 10:32, 11:32, 12:32, 13:32, 14:32, 15:32, 16:32, 17:32
日曜 09:32, 10:32, 11:32, 12:32, 13:32, 14:32, 15:32, 16:32, 17:32
月曜 09:32, 10:32, 11:32, 12:32, 13:32, 14:32, 15:32, 16:32, 17:32
火曜 09:32, 10:32, 11:32, 12:32, 13:32, 14:32, 15:32, 16:32, 17:32
```

月曜と火曜は開催がある場合のみ更新対象です。毎時32分にしているのは、JRA側の発表やオッズ更新直後の混雑を避けるための余裕です。

## 現在の制約

- オッズは印決定には使っていません。表示用に人気順位だけを公開します。
- 馬場状態、天候、枠順傾向は使っていません。
- 騎手、斤量、性齢は取得しますが、現行スコアには使っていません。
- 血統は父名と母の父名を静的CSVに照合します。牝系、調教、厩舎コメント、新聞印は使っていません。
- JRA公式HTMLの構造変更に弱いです。
- 着差の解釈は単純な括弧内数値の抽出です。
- スコアは予測確率ではなく、順位付け用の指数です。
